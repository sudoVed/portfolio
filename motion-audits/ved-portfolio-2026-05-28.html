🔴 Critical

Blur-everywhere entrances — identical parameters across all Act Two elements

A single GSAP fromTo call applies { y: 42, opacity: 0, filter: "blur(8px)" } identically to every .reveal element: hero copy, profile frame, both section headings, all four skill rows, all three project panels, and the contact section. That's 12+ elements sharing the same entrance values, which is the textbook blur-everywhere slop pattern (flag threshold: ≥4 elements with identical parameters).

The result: the hero heading "VED" lands with the same weight as a skill tag. A section heading has the same entrance as a project panel. There's no motion hierarchy — everything is equally important and equally bland. The design language establishes spatial depth; the animation should reinforce it.

Fix: Differentiate by role. Hero mark (.hero-mark): longer duration (1.1s), larger translateY (60px), stronger blur (14px) — it's a rare, marquee moment. Skill rows: shorter (0.6s), horizontal slide-in (translateX(-8px)), no blur — stat blocks feel tabular, not floating. Project panels: keep blur but reduce to 6px, translateY 24px. Section headings: clip-path reveal (vertical wipe) instead of blur. Contact section: opacity only, 0.5s — quietest moment on the page.

🟡 Important

CSS variable updates on every pointermove — cascade recalculation on tilt

element.style.setProperty("--tilt-x", ...) and setProperty("--tilt-y", ...) fire on every pointermove event (every ~16ms). CSS variable updates invalidate style recalculation through the entire subtree — the project panel's children (h3, tags, link button, visual) all recalculate on every frame. With three project panels potentially in view, that's a significant compute budget for an ambient effect.

The spotlight --spot-x / --spot-y variables feeding the radial-gradient can stay as CSS variables (gradient doesn't trigger child layout). The rotation tilt should be applied directly to element.style.transform.

Fix: In onMove, replace setProperty("--tilt-x", ...) and setProperty("--tilt-y", ...) with a single direct element.style.transform = `perspective(1200px) rotateX(...) rotateY(...)`. In onLeave, reset with element.style.transform = "perspective(1200px) rotateX(0deg) rotateY(0deg)".

Issues to Address

🟡 Important

Skill rows enter simultaneously — no stagger reinforces the stat-block reading order

The four skill rows (Systems & Robotics, ML & AI, Software & Infrastructure, Tools & Workflow) all enter the viewport at the same ScrollTrigger threshold with identical animation parameters. This makes the "stat block" design metaphor feel like a dump rather than a reveal. The D&D-inspired layout wants to be read top-to-bottom; simultaneous entrance undermines that directionality.

A sequential stagger (80–100ms per row) creates the feeling of entries resolving one after another — like a stat table populating. Combine with a horizontal slide-in (translateX(-8px) → 0) rather than vertical float, since the rows are horizontal data structures.

Fix: In the GSAP reveal loop, detect skill rows and apply per-index delay: delay: index * 0.09 (where index is derived from the row's position in the skill table). Or use CSS custom property --row-index on each row and CSS animation-delay: calc(var(--row-index) * 90ms) with animation-fill-mode: backwards to prevent pre-flash.
